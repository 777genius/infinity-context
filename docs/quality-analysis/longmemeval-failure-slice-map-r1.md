# LongMemEval 100-case failure slice map r1

Analysis date: 2026-07-19. This is a read-only analysis of the existing baseline; no benchmark was rerun and no provider was called.

## Decision summary

The baseline is trustworthy as a completed retrieval/context run, but it is not an answer-accuracy result. It scored whether the rendered context contained every official LongMemEval answer-session id. The aggregate result is 61/100 (0.61), with 39 `missing_expected_terms` failures and no infrastructure failures or forbidden-term leaks.

The dominant observed problem is evidence-package incompleteness:

- 10 failures retrieved 8-20 items but rendered none of the answer sessions (22 answer sessions missed in total).
- 29 failures rendered some answer sessions but not all (43/90 answer sessions covered).
- Every knowledge-update failure covered exactly one of two required sessions.
- Multi-session and temporal cases account for 32/39 failures and cover only 23/65 and 15/35 required answer sessions, respectively.

The five slices below form a strict partition of all 39 failures. They deliberately avoid the active `context_source_sibling*` r6b code and tests. The recommended first changes are paired-evidence completeness and distinct-set completeness; the heterogeneous zero-hit group has the largest causal uncertainty and should be instrumented before broad tuning.

## What this baseline measures

The shard reports declare `evaluation_mode: retrieved_expected_terms`. For official LongMemEval rows, `public_benchmark.py` uses `answer_session_ids` as `expected_terms` and prefixes each ingested session document with its session id. After `/v1/context`, the evaluator searches the returned `rendered_text` and item text/source fields for those ids.

Consequences:

- `answer_preview` is the dataset's gold answer preview, not an answer generated by Infinity Context.
- A pass proves that all labeled answer sessions survived into rendered evidence; it does not prove that an answerer could derive the correct value.
- A failure proves that at least one labeled answer session was absent from rendered evidence. It does not by itself distinguish ingest, query planning, collection, ranking, hydration, deduplication, budgeting, or rendering.
- Session ids are valid post-hoc labels for evaluation, but must never become query, ranking, selection, or answer inputs.

| Mode | Cases | Direct report evidence | Supported conclusion | Not yet proved |
|---|---:|---|---|---|

These modes describe observable evidence coverage, not root causes. Question wording supports the proposed implementation boundaries, while the report coverage values provide the hard failure evidence.

## Shared acceptance and anti-overfitting contract

Every slice must satisfy all of these before its focused case gate is meaningful:

1. Production selection accepts only the user question, authorized scope, candidate text, source identity, and canonical temporal metadata. It has no parameter or lookup for gold answers, `answer_session_ids`, `expected_terms`, case ids, or marker hashes.
2. Application tests assert on visible, source-labeled semantic evidence in `ContextBundle.rendered_text` and structured items. They do not create `answer_*` marker tokens or declare success from an opaque marker alone.
3. Mirrored fact/chunk evidence from one canonical source cannot satisfy two required roles or inflate set cardinality.
4. Existing evidence-only prompt safety, visibility, source labeling, deterministic ordering, and token-budget behavior remain green.
5. The targeted public-benchmark command is only a post-hoc regression gate. Omit `--api-url` so it uses the local in-process test profile with SQLite and provider adapters disabled. Do not run medium/full LongMemEval in a slice.
6. Before a focused run, the staged `$LONGMEMEVAL_DATASET` must hash to `d6f21ea9d60a0d56f34a05b609c79c88a451d2ae03597821ea3d5a9678c3a442`.

### Slice 1 — paired evidence requirements

Cases (10; all cover 1/2 answer sessions):

`031748ae`, `45dc21b6`, `6071bd76`, `6aeb4375`, `71315a70`, `gpt4_1916e0ea`, `gpt4_1d4ab0c9`, `gpt4_1e4a8aeb`, `gpt4_4929293a`, `gpt4_fa19884c`.

Boundary:

- Detect two-role evidence requirements from question-only state-transition, comparison, and two-endpoint language.
- Reserve distinct canonical sources for both roles before generic support consumes the budget.
- This slice does not perform temporal arithmetic, distinct-set counting, query fanout, or source-sibling expansion.

Proposed owned paths:

- `packages/infinity_context_core/infinity_context_core/application/context_requirement_coverage.py`
- `packages/infinity_context_core/infinity_context_core/application/context_requirement_guard.py`

- `tests/unit/test_context_paired_evidence_requirements.py` (new)

Rendered-evidence assertions:

- Before/now fixtures render one source-labeled item for the earlier state and a different canonical source for the current state.
- Interval/first-event fixtures render both named endpoints; duplicating one source cannot satisfy both slots.
- If one role is unavailable, diagnostics say which question-derived role is missing without exposing raw text or gold labels.

## Uncertainty and stop conditions

- The shard reports omit retrieved text, ranks, scores, decomposed queries, per-stage candidate ids, and source refs for these official LongMemEval cases. Opaque `chunk_*` item ids cannot identify the loss stage.
- The aggregate is one run. It provides no repeat-run variance or confidence interval.
- Answer-session labels can be broader than the minimal answer span, and marker coverage can pass even if the evidence excerpt is semantically insufficient. Focused application tests must therefore assert rendered semantic evidence as well as the post-hoc session gate.
- A slice should stop and be reclassified if stage diagnostics show that its support sessions never entered that slice's owned seam. Do not compensate downstream for an upstream miss.
- No medium/full benchmark should run until the slice's deterministic rendered-evidence tests and its 5-10 exact-case local gate are green, and no broader run should be used to hide a failed focused gate.
